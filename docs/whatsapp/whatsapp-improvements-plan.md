# خطة التحسينات لنظام WhatsApp في منصة CRM

**التاريخ:** 27 يناير 2026  
**الحالة:** قيد التنفيذ

---

## ملخص المراجعة

بعد مراجعة شاملة لوثائق WhatsApp Business API من Meta، تم تحديد **التحسينات الحرجة** التالية لضمان التوافق الكامل مع تعليمات Meta واستغلال جميع الميزات المتاحة.

---

## التحسينات المطلوبة

### 1. استخدام Marketing Messages API Endpoint ⭐⭐⭐ (حرج)

**المشكلة الحالية:**
- نستخدم `/messages` endpoint لجميع الرسائل

**التحسين المطلوب:**
- **استخدام `/marketing_messages` endpoint** لجميع الرسائل التسويقية
- هذا يفعّل automatic delivery optimizations من Meta
- يوفر performance benchmarks و tailored recommendations

**الفائدة:**
- زيادة معدلات القراءة والنقرات (أثبتت الاختبارات تحسن كبير)
- وصول أفضل للجمهور المستهدف
- تقارير حصرية عن الأداء

**الكود المطلوب تعديله:**
```typescript
// ❌ قديم
const url = `https://graph.facebook.com/${API_VERSION}/${PHONE_NUMBER_ID}/messages`;

// ✅ جديد للرسائل التسويقية
const url = `https://graph.facebook.com/${API_VERSION}/${PHONE_NUMBER_ID}/marketing_messages`;
```

### 2. تحسين نظام القوالب (Templates) ⭐⭐⭐ (حرج)

#### 2.1 استخدام Named Parameters بدلاً من Positional

**المشكلة الحالية:**
- نستخدم `{{1}}`, `{{2}}` (positional parameters)

**التحسين المطلوب:**
- استخدام `{{patient_name}}`, `{{appointment_date}}` (named parameters)
- أكثر وضوحاً وأقل عرضة للأخطاء
- يمكن ترتيب المتغيرات بأي شكل عند الإرسال

#### 2.2 إضافة Time-to-Live (TTL) للرسائل الحساسة للوقت

**الاستخدام:**
- رسائل OTP (صالحة لـ 5-10 دقائق فقط)
- عروض محدودة الوقت (تنتهي في وقت محدد)
- رسائل التذكير (لا قيمة لها بعد الموعد)

**الفائدة:**
- تجنب إرسال رسائل غير ذات صلة أو متأخرة
- تحسين تجربة المستخدم
- تقليل معدلات الحظر

#### 2.3 مراقبة Quality Rating للقوالب

**المطلوب:**
- إضافة webhook subscription لـ `message_template_quality_update`
- عرض Quality Rating في صفحة إدارة القوالب:
  * 🟢 GREEN: جودة عالية
  * 🟡 YELLOW: جودة متوسطة (تحذير)
  * 🔴 RED: جودة منخفضة (خطر الإيقاف)
- إشعار تلقائي للمدير عند تدهور الجودة

---

### 3. تحسين نظام Webhooks ⭐⭐⭐ (حرج)

#### 3.1 إضافة Webhook Fields المهمة

**الحالي:**
- نستقبل `messages` فقط

**المطلوب إضافته:**
| Field | السبب |
|-------|-------|
| `message_template_status_update` | معرفة حالة القوالب (معتمد/مرفوض) فوراً |
| `message_template_quality_update` | مراقبة جودة القوالب |
| `account_alerts` | تنبيهات حدود الإرسال والملف التجاري |
| `business_capability_update` | تغييرات في قدرات الحساب |

#### 3.2 تحسين معالجة Webhooks

**المطلوب:**
1. **التحقق من `hub.verify_token`** عند التسجيل
2. **إرجاع HTTP 200 دائماً** (حتى لو حدث خطأ داخلي)
3. **معالجة Webhooks بشكل async** (عدم حجب الرد)
4. **تخزين Webhooks الفاشلة** لإعادة المعالجة لاحقاً
5. **تجنب التكرار** (Meta قد ترسل نفس webhook مرتين)

#### 3.3 إضافة Mutual TLS (mTLS)

**الفائدة:**
- أمان إضافي للـ webhooks
- بديل أفضل من تقييد IP addresses

---

### 4. إضافة نظام Message Queue ⭐⭐ (مهم)

**المشكلة الحالية:**
- إرسال الرسائل بشكل متزامن (synchronous)
- قد يسبب timeout أو ضغط على السيرفر

**التحسين المطلوب:**
- استخدام Message Queue (مثل Bull أو BullMQ)
- معالجة الرسائل بالتتابع
- retry mechanism للرسائل الفاشلة
- rate limiting تلقائي

**الفائدة:**
- تجنب الضغط على السيرفر
- ضمان إرسال جميع الرسائل
- سهولة مراقبة الأداء

---

### 5. تحسين معالجة الأخطاء ⭐⭐ (مهم)

**المطلوب:**
1. **تسجيل Error Codes** من Meta API
2. **معالجة الأخطاء الشائعة:**
   - `131049`: لا يمكن إرسال رسائل تسويقية للولايات المتحدة
   - `131026`: قالب غير معتمد أو متوقف
   - `131047`: تجاوز حد الرسائل
   - `131051`: رقم المستخدم محظور أو غير صحيح

3. **إشعارات تلقائية** عند حدوث أخطاء حرجة

---

### 6. إضافة ميزات متقدمة ⭐ (مستقبلية)

#### 6.1 Rich Media Support
- دعم GIFs في الرسائل التسويقية
- Automatic creative optimizations (تحريك الصور تلقائياً)

#### 6.2 Insights & Analytics
- تقارير أداء الحملات
- معدلات القراءة والنقرات
- مقارنة الأداء مع الأعمال المماثلة

#### 6.3 User Preferences
- احترام تفضيلات المستخدم للرسائل التسويقية
- webhook لـ `user_preferences` لتحديث القاعدة تلقائياً
