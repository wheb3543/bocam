## 1️⃣ هندسة عاملي الخدمة الثنائي (Dual Service Workers Architecture)

من أجل ضمان كفاءة التشغيل وعزل الصلاحيات، لا تعتمد المنصة على عامل خدمة واحد؛ بل تستخدم **مبدأ العزل الثنائي**:

### أ. عامل خدمة بوابة المريض والزوار العامة (`sw.js`)
* **الملف المسؤول:** `/client/public/sw.js`
* **مستوى الكاش الرئيسي:** `sgh-public-v1`
* **مستوى الكاش المؤقت لبيئة التشغيل:** `sgh-public-runtime-v1`
* **المسارات المستهدفة:** المسار الرئيسي `/` وبوابة المرضى `/patient`. ويتم تجاهل مسارات لوحة التحكم الإدارية `/admin` تماماً لتفادي تداخل الكاش.

### ب. عامل خدمة لوحة التحكم الإدارية (`sw-admin.js`)
* **الملف المسؤول:** `/client/public/admin/sw-admin.js`
* **مستوى الكاش الرئيسي:** `sgh-admin-v2`
* **مستوى الكاش المؤقت لبيئة التشغيل:** `sgh-admin-runtime-v2`
* **المسارات المستهدفة:** مسارات لوحة التحكم الإدارية `/admin` بشكل حصري.

```
                         ┌─────────────────────────────────┐
                         │      SGH CRM Portal Entry       │
                         └────────────────┬────────────────┘
                                          │
                  ┌───────────────────────┴───────────────────────┐
                  ▼                                               ▼
     ┌─────────────────────────┐                     ┌─────────────────────────┐
     │  Patient/Public Routes  │                     │   Admin Portal Routes   │
     │      (e.g., / )         │                     │      (e.g., /admin)     │
     └────────────┬────────────┘                     └────────────┬────────────┘
                  │                                               │
                  ▼                                               ▼
     ┌─────────────────────────┐                     ┌─────────────────────────┐
     │  Public Service Worker  │                     │   Admin Service Worker  │
     │       (sw.js)           │                     │      (sw-admin.js)      │
     └─────────────────────────┘                     └─────────────────────────┘
```

## 2️⃣ دورة حياة الخدمة (Service Worker Lifecycle)

يمر عاملا الخدمة في المتصفح بثلاث مراحل دورة حياة قياسية ومؤمنة:

### 1. مرحلة التثبيت (Install Event)
* عند تحميل الصفحة لأول مرة، يتم تسجيل الـ Service Worker وفتح مخزن الكاش المخصص له (CACHE_NAME).
* يتم سحب الملفات الأساسية المحددة في مصفوفة التخزين المسبق (**Precache URLs**) مثل الـ `manifest.json` والأيقونات والخطوط الأساسية وصفحة الهبوط دون اتصال.
* يتم استخدام خيار `Promise.allSettled` لتجنب فشل عملية التثبيت بأكملها في حال تعذر تحميل ملف فردي مفقود.
* ينتهي الحدث بـ `self.skipWaiting()` لإجبار عامل الخدمة الجديد على الانتقال فوراً للمرحلة القادمة دون انتظار إغلاق الصفحات النشطة.

### 2. مرحلة التنشيط (Activate Event)
* يتم فحص جميع مخازن الكاش القديمة في المتصفح.
* يقوم السيرفر بفلترة المخازن القديمة التي تبدأ ببادئة التطبيق (مثل `sgh-public-` أو `sgh-admin-`) ومسحها تلقائياً عند ترقية الإصدار، مع الاحتفاظ بالمخازن النشطة الحالية لتجنب مسح بيانات تشغيل المستخدم.
* ينتهي بـ `self.clients.claim()` لفرض السيطرة الفورية على كافة صفحات الويب المفتوحة.

### 3. مرحلة اعتراض الطلبات (Fetch Event)
* يعترض عامل الخدمة جميع طلبات الشبكة الصادرة من النطاق نفسه (**same-origin**)، ويتم استبعاد طلبات الـ CORS والـ CDNs الخارجية لتجنب مشاكل تعارض سياسات الأمان أو توقف الاتصال.

## 3️⃣ استراتيجيات التخزين المؤقت (Caching Strategies)

تطبق المنصة استراتيجيات تخزين هجينة ذكية مخصصة حسب نوع البيانات وطبيعة الطلب:

### 📈 أ. استراتيجية الشبكة أولاً مع بديل محلي (Network-First with Cache Fallback)
* **تُطبق على:** طلبات التوجيه والتنقل العامة للـ HTML (Navigation Requests).
* **آلية العمل:** يقوم المتصفح بمحاولة جلب أحدث نسخة من الصفحة عبر الشبكة أولاً؛ وإذا كان الاتصال متاحاً، يتم إرجاع الصفحة وتحديث نسختها المخزنة في الكاش المؤقت (Runtime Cache). في حال انقطاع الشبكة، يتم فحص الكاش وعرض النسخة المخزنة مسبقاً، وإذا تعذر ذلك، يتم التوجيه لصفحة الطوارئ دون اتصال.
* **السبب:** ضمان حصول المرضى والموظفين على البيانات الطبية المحدثة واللحظية دوماً عند وجود اتصال، مع توفير واجهة بديلة دافئة في حال انقطاع الشبكة المفاجئ.

### 📦 ب. استراتيجية الكاش أولاً (Cache-First)
* **تُطبق على:** المكونات الثابتة (Static Assets) مثل ملفات الـ CSS والـ JS المصنفة، الصور التعريفية للأطباء، شعار المستشفى، والأيقونات.
* **آلية العمل:** يتم فحص الكاش أولاً؛ وإذا وُجد الملف، يُرجع فوراً دون إرسال أي طلب للسيرفر مما يقلل استهلاك باقة الإنترنت واستهلاك السيرفر. في حال عدم وجوده، يتم جلبه من الشبكة وتخزينه للاستخدامات المستقبلية.

### 🛑 ج. استبعاد واجهات التطبيق البرمجية (API Pass-through - Bypassing)
* **تُطبق على:** جميع مسارات واجهة التطبيق البرمجية (`/api/`).
* **آلية العمل:** يُمنع منعاً باتاً تخزين طلبات الـ API أو استعلامات tRPC في كاش المتصفح الافتراضي للـ Service Worker. يتم توجيهها للشبكة مباشرة؛ وفي حال عدم وجود اتصال، تعيد الواجهة كائن JSON يوضح خطأ في الاتصال دون خنق المتصفح.

## 4️⃣ تجربة عدم الاتصال والـ Fallbacks (Offline Experience & Fallbacks)

في حال فقدان كامل للاتصال بشبكة الإنترنت، يتصرف النظام بمرونة لضمان ثبات تجربة المستخدم:

* **بوابة المريض:** عند فقدان الاتصال، يتم توجيه المريض تلقائياً إلى صفحة عدم الاتصال المخصصة (`/offline`). تعرض هذه الصفحة رسالة ودية توضح انقطاع الاتصال مع عرض معلومات تواصل المستشفى المخزنة محلياً ورقمه المجاني الموحد (**8000018**)، بالإضافة لعرض آخر البيانات المحفوظة في كاش التصفح لتقليل حدة الإزعاج.
* **لوحة التحكم الإدارية:** في حال انقطاع الإنترنت لدى موظفي خدمة العملاء أو مسؤولي الـ CRM، يمنع السيرفر تشنج لوحة التحكم ويقوم بتوجيه المسار لصفحة `/admin` المخزنة محلياً مع عرض إشعار علوي عائم (Toast Notification) يوضح العمل بـ **"وضع عدم الاتصال"**، وحفظ آخر الاستعلامات والعمليات محلياً (stale data).

## 5️⃣ تحديثات التطبيق (App Updates)

لضمان عدم بقاء المراجعين وموظفي الاستقبال على نسخ قديمة ومخزنة من الواجهات البرمجية والتصميمية عند نشر تحديث جديد للمنصة:

1. **طريقة تدمير الكاش التلقائي (Cache Busting):**
   - بمجرد بناء المنصة في الإنتاج، يتم تغيير ترويسة الإصدار لملف الكاش (`CACHE_NAME = 'sgh-admin-v2'`, `sgh-public-v2`).
   - عند تحميل النسخة الجديدة، يتحقق حدث `activate` من وجود أسماء كاش قديمة لا تطابق الاسم الجديد، ويقوم بحذف الكاش القديم بالكامل برمجياً.
2. **استخدام `skipWaiting` و `clients.claim`:**
   - يتيح ذلك للـ Service Worker الجديد التنشيط والتحكم بالصفحات المفتوحة فوراً وتحديث الملفات دون الحاجة لإغلاق المتصفح وإعادة فتحه يدوياً من قبل المستخدم.
3. **تنبيهات التحديثات الإجبارية والاختيارية (Update Modals):**
   - يرتبط التطبيق بنظام فحص ذكي مدمج (`server/_core/updateChecker.ts`) للتحقق من سلامة وإصدار واجهات التطبيق ومطابقتها، وإظهار نافذة منبثقة للمسؤولين في حال صدور تحديث إجباري هام لتنظيف الكاش فوراً وإعادة تحميل الصفحة لضمان سلامة العمليات وتوافق البيانات.
