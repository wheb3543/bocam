# خارطة طريق وإستراتيجية الإصلاح الجذري لأخطاء TypeScript (TypeScript Fix Roadmap)

تم إعداد هذه الوثيقة كدليل عملي وخطة تنفيذية شاملة لحل **679 خطأ برمجياً في 52 ملفاً** بشكل جذري واحترافي، دون كسر أي منطق تشغيلي قائم، ومع إتمام أي ميزات برمجية غير مكتملة تم كشفها بعد إزالة الـ `any`.

---

## 📌 تعليمات صارمة ومبادئ توجيهية أثناء التنفيذ (Strict Guidelines for Execution)
يجب على أي مهندس برمجيات أو وكيل ذكاء اصطناعي (Agent) يلتزم بتنفيذ هذه الخطة، التقيد التام بالقواعد التالية:

1. **ممنوع منعاً باتاً كبت الأخطاء (No Suppressions):**
   * يُحظر استخدام `as any` أو `as unknown` أو الالتفاف عبر `// @ts-ignore` أو `// @ts-nocheck` لإخفاء الأخطاء. يجب حل الخطأ برمجياً من جذره بتصحيح التعريفات.
2. **الاعتماد على مصدر موحد للأنواع (Single Source of Truth):**
   * يجب أن تتطابق الواجهات الأمامية (Frontend Interfaces) تماماً مع الواجهات الخلفية وقاعدة البيانات (tRPC/Prisma/Drizzle). لا يُسمح بتكرار تعريف الكيانات محلياً في الصفحات.
3. **التعامل الآمن مع القيم الفارغة (Null/Undefined Safety):**
   * يجب افتراض أن أي حقل قادم من قاعدة البيانات قد يكون `null` أو `undefined`. يجب استخدام معاملات الحماية مثل الاختيار الاختياري (`?.`) ومشغل الدمج الافتراضي (`??`) دائماً.
4. **محولات البيانات الصريحة والتطهير (Explicit Data Mapping):**
   * عند قراءة التواريخ أو المصفوفات من الخادم، يجب تمريرها أولاً عبر دالة تحويل ومواءمة (Mapper) للتأكد من صيانة الأنواع وتحويل الـ Strings والـ Nulls إلى كائنات `Date` أو قيم بديلة آمنة قبل تمريرها للمكونات.
5. **الاختبار المستمر والتراكمي (Continuous Validation):**
   * بعد الانتهاء من كل مرحلة، يجب تشغيل أمر فحص الأنواع للتأكد من انخفاض عدد الأخطاء وعدم إدخال أخطاء جديدة:
     ```bash
     pnpm tsc --noEmit
     ```
